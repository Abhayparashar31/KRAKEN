Google Dorking

A field guide to advanced search-operator reconnaissance (OSINT)

Google dorking is the practice of using advanced search operators to surface specific, often overlooked information that a search engine has already indexed. It's a foundational OSINT and security-research skill — used by penetration testers to scope authorized engagements, by security teams to check their own exposure, and by researchers to filter noise out of the open web.

Authorized use only
01 / Core Operators

The syntax you'll actually use

These operators work on Google (most have equivalents on Bing and DuckDuckGo). Combine several in one query for precision — that's where dorking gets powerful.

02 / Illustrative Patterns

Common reconnaissance categories

These are the categories most public OSINT guides and bug-bounty checklists reference — shown here at the pattern level so you understand why each works, not as a plug-and-play attack script.

Exposed documents

Sensitive files left publicly indexed

site:target.com filetype:pdf OR filetype:xlsx "confidential"

Finds spreadsheets or PDFs containing a sensitivity marker that were never meant to be public-facing but got crawled anyway.

Directory listings

Open, unindexed folders

intitle:"index of" "parent directory" site:target.com

Surfaces raw directory listings — a classic sign of a misconfigured web server exposing its file structure.

Login portals

Mapping authentication surfaces

inurl:login site:target.com

Helps a tester (with authorization) enumerate which subdomains expose admin or user login pages, for scoping — not for credential attempts.

Configuration leaks

Environment and config artifacts

site:target.com filetype:env OR filetype:yml "DB_PASSWORD"

Targets configuration file extensions that sometimes get deployed to production by mistake.

Error messages

Stack traces & verbose errors

site:target.com intext:"Warning: mysql_connect()"

Verbose server errors indexed by search engines often hint at outdated software versions or misconfigurations.

Cloud storage

Misconfigured buckets

site:s3.amazonaws.com "target-name"

Looks for cloud storage buckets named after or referencing an organization that may have been left with public read access.

03 / Field Notes

Tips for getting good results

01

Stack operators, don't scatter queries

A single query combining site: + filetype: + a quoted phrase filters harder than three loose searches.

02

Use the Google Hacking Database (GHDB) as a reference

Exploit-DB's GHDB catalogs dork patterns by category and CVE — useful for understanding what's already publicly documented rather than reinventing queries.

03

Pair with passive recon tools

Tools like Shodan, Censys, and crt.sh complement dorking by indexing infrastructure metadata search engines don't crawl (open ports, TLS certs).

04

Narrow with date and region operators

Use Google's "Tools" menu (Any time / region) after your query to cut down on stale or irrelevant indexed pages.

05

Cross-check with cache

If a page has since been taken down, web archives (e.g. the Wayback Machine) can show what was indexed at the time — useful for timeline reconstruction in an investigation.

06

Run dorks against your own org first

The single highest-value use of dorking for most teams is turning it inward: finding what your own company has accidentally exposed before someone else does.

04 / Scope & Ethics

Where the line is

Read before you search

Google dorking only touches content a search engine has already crawled and publicly indexed — it isn't a hacking technique on its own. What you do with what you find is what determines legality and ethics:

  • Authorization matters. If you're testing an organization's exposure, get written scope/permission first (a bug bounty program, a signed pentest agreement, or it's your own infrastructure).
  • Finding ≠ accessing. Discovering an exposed login page or file listing is different from logging in, downloading, or modifying anything you find — the latter can cross into unauthorized access under laws like the U.S. CFAA or the UK Computer Misuse Act.
  • Report, don't exploit. If you stumble on someone else's real exposed data outside an authorized engagement, the responsible move is a good-faith disclosure to the organization, not further probing.
  • Respect robots.txt and ToS. Some sites explicitly disallow automated scraping of indexed content even when technically reachable.
