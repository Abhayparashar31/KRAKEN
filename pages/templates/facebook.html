Facebook

Query syntax for Meta Content Library and the Graph API

Unlike Google or X, Facebook doesn't expose a public "advanced search" bar with site:-style operators. Instead, precise querying happens through two official Meta surfaces: the Meta Content Library & API (Boolean search over public posts, for qualified researchers) and the Graph API (structured field/edge queries, for app developers with the right permissions). This guide covers the real, documented syntax for both.

Authorized use only
01 / Core Operators & Syntax

The two query systems

Meta Content Library uses Boolean search operators. The Graph API uses URL parameters and field-expansion syntax instead of search operators — you're querying specific objects and edges, not free text. Both are shown below.

02 / Sample Queries

Worked examples for both systems

Content Library queries run against the search/facebook_posts endpoint; Graph API examples run against graph.facebook.com with a valid access token and the required permissions.

Content Library · Boolean AND

All terms must be present

q = "election & misinformation & 2024"

Matches posts containing all three terms — the default building block for narrowing a research query.

Content Library · Boolean OR

Either term may appear

q = "flood | wildfire"

Casts a wider net across two related disaster-related keywords in a single search.

Content Library · NOT + grouping

Excluding a term, precedence controlled

q = "(vaccine | booster) - satire"

Finds posts about vaccines or boosters while excluding anything also tagged satire — parentheses force the OR to resolve before the NOT is applied.

Graph API · field expansion

Page posts with nested engagement data

GET /{page-id}?fields=name,posts.limit(10){message,created_time,comments.summary(true),reactions.summary(true)}

Pulls a page's recent posts along with comment and reaction totals in one round trip instead of N+1 separate calls.

Graph API · time-range filter

Posts within a specific window

GET /{page-id}/posts?since=2024-01-01&until=2024-01-31&limit=100

Restricts results to a date range — the standard pattern for incremental or historical pulls.

Graph API · Insights metrics

Aggregated Page performance

GET /{page-id}/insights?metric=page_impressions,page_engaged_users&period=day&since=2024-01-01&until=2024-01-31

Returns pre-aggregated daily metrics rather than raw post data — far cheaper than computing engagement client-side.

03 / Pro Tips

Getting precise, efficient results

01

Remember parse order in Boolean queries

Content Library always evaluates NOT first, then AND, then OR — left to right for equal precedence. Use parentheses whenever the default order isn't what you mean.

02

Only single keywords, no phrases or wildcards

Content Library search doesn't support quoted phrases or * wildcards — build intent through combinations of single terms and operators instead.

03

Use field expansion instead of chained calls

Nesting edges with { } in one Graph API request avoids the N+1 problem and counts as a single call against your rate limit.

04

Prefer summary(true) over fetching every record

When you only need a count (comments, reactions), the summary modifier returns aggregates without pulling every individual object.

05

Always paginate with the provided cursor

Use the paging.next URL Facebook returns rather than manually incrementing offsets — manual pagination breaks silently as data changes.

06

Batch related Graph API calls

Up to 50 operations can be bundled into one batch request, and later operations can reference earlier results via JSONPath — useful for dependent lookups like "user → their posts."

07

Request only the fields you need

The Graph API returns nothing by default until fields are named — build reusable field templates for your common query shapes rather than over-fetching.

04 / Guidance & Access

Where the line is

Read before you query
  • Content Library access is gated. It's built for qualified academic and public-interest researchers, not general public search — Meta requires an application and approval before you get API or tool access.
  • Graph API access needs real permissions. Reading anything beyond public Page data requires an access token scoped to that data, and user-level data requires the person's consent through app review — you can't query private profiles or friend lists at will.
  • Never expose access tokens client-side. Route Graph API calls through a server-side proxy and rotate tokens; a leaked token is equivalent to leaked credentials.
  • Respect platform terms and data-use limits. Both systems come with usage restrictions (rate limits, retention rules, redistribution limits) — read the disclosures for the system you're using before building on top of it.
